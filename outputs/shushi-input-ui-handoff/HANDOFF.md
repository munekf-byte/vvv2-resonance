# 【真打吉宗担当への引き継ぎ】収支入力UI 移植パッケージ

このパッケージは TGR Resonance プロジェクトで確定した「収支入力ダッシュボード」を、真打吉宗プロジェクトに **既存コードを壊さない形で** 移植するための自己完結ハンドオフです。

---

## ⚠ 絶対遵守の作業順序

**Step 1 → 確認 → Step 2 の順で進めてください。Step 1 完了前に Step 2 へ進まないこと。**

### Step 1: 新規UIをまず単独で実装する（既存は触らない）

- このパッケージの内容のみを使って、新しい収支入力ダッシュボードを実装する。
- 既存の収支関連コード（もし真打吉宗プロジェクト内に類似機能があれば）には **一切手を加えない**。
- 新ファイルとして並列に作り、新規ルート or テスト画面で動作確認する。
- ユーザーに「Step 1 完了、動作確認OK」の報告を上げる。

### Step 2: 既存スキル・既存コードを本パッケージ準拠で更新する

- Step 1 で動作確認が取れた **後** に、このパッケージ（特に §3 SKILL.md 仕様書）を読み込み、真打吉宗プロジェクトの既存スキル・既存収支関連コードを **自律的に** 本パッケージ準拠へ書き換える。
- 書き換え方針はユーザーに確認しなくてよい（パッケージ自体が方針なので）。
- ただし、機種固有のカスタマイズが必要な箇所（保存ボタン色、ヘッダーラベル等）は §6 のカスタマイズポイント表に従う。

## 🚀 Step 1: 最短実装手順（5ステップ）

### 1. ShushiData 型を追加

`templates/types.ts` の内容を真打吉宗プロジェクトの型定義ファイル（`src/types/index.ts` 相当）の末尾に追加。

```typescript
export interface ShushiData {
  coinRate: number;
  handCoins: number | null;
  cashInvestK: number | null;
  exchangeCoins: number | null;
  totalGames: number | null;
}
```

既に類似の型がある場合は、**新規追加のみ** とし、既存型は触らない（Step 1 では）。  
→ 名前衝突する場合は `ShushiDataV2` 等で一時的に並走させる。

### 2. コンポーネント本体を配置

`templates/ShushiEditDashboard.tsx` を以下にコピー:

```
src/components/<機種ディレクトリ>/ShushiEditDashboardV2.tsx
```

※ Step 1 中は **V2 サフィックス** を付けて、既存があれば衝突回避。

### 3. import パスを調整

ファイル先頭の `import type { ShushiData } from "@/types";` を、真打吉宗プロジェクトのエイリアスに合わせる。それ以外は **一切変更しない**。

### 4. 親画面に組み込む

`templates/integration-example.tsx` を参考に、テスト用のFABボタン + state を追加して新規UIを起動。

### 5. 動作確認

確認項目:
- [ ] フルスクリーンモーダルが正しく開閉する
- [ ] 4セクション（レート/投資/交換/トータルG数）が色分けされて表示
- [ ] 投資合計が「投資>0」のときだけ赤バナーで出る
- [ ] 最終収支が緑/赤で正しく計算される（交換 − (手持ち + 現金k×レート)）
- [ ] iOS Safari と Android Chrome で safe-area が効いている
- [ ] キーボード閉じ時にフッターが隠れない

**ここまで完了したら、ユーザーに報告。Step 2 は報告後。**

## 🔁 Step 2: 既存スキル・コード更新（Step 1 完了後）

### 2-A. 真打吉宗プロジェクトの既存「収支入力」スキルがあれば、本パッケージ §3 SKILL.md の内容で **置き換える**

- 既存スキルファイル（`.claude/skills/*収支*` 等）を本パッケージ準拠に書き換え
- セクション順序・色定義・アンチパターンを丸ごと採用

### 2-B. 既存の収支入力コンポーネントを V2 へ移行

1. Step 1 で配置した `ShushiEditDashboardV2.tsx` を `ShushiEditDashboard.tsx` にリネーム（既存があれば置換）
2. 既存呼び出し側を新コンポーネント参照に切替
3. 旧 ShushiData 型と新型に差分があれば、新型へマイグレーション（古いフィールドは保持してフォールバックで読む）
4. `localStorage` に旧 totalG キーが存在する場合は読み取り専用フォールバックで残す（最低 30 日）

### 2-C. 既存の集計画面に G数入力欄がある場合は削除

- 「TOTAL G数」を集計画面で別途入力させていた場合、その入力欄は削除する
- 集計画面は `shushi.totalGames` を参照する側に変更
- 「収支入力が真実の単一ソース」原則を守る

### 2-D. 横展開チェックリストを最後に流す

§7 のチェックリストを上から順に確認し、すべて ✓ になったらユーザーに完了報告。
