# §1 仕様書

## 1. モード定義と不転落Stateモデル

本機の核心は **「天国モードに到達するまでモード転落がない」** という点。これを「Block（初当たり間の区間）」を跨ぐ状態遷移として定義する。

### 1.1 滞在モード定義

| モード (id) | 天井G数 | 特徴 |
|---|---|---|
| MODE_A      | 600G | 基本モード |
| MODE_B      | 600G | 100G/300G/500G がチャンス |
| MODE_C      | 500G | 天井が低い。200G/400G がチャンス |
| MODE_CH     | 600G | 規定G数当選期待度が高い |
| MODE_PRE    | 300G | 天国への足掛かり。150G/200G がチャンス |
| MODE_HEAVEN | 100G | 100G以内に必ず当選 |

### 1.2 ブロック間遷移ロジック

- **CZ失敗時**: Block(N) の事後確率を Block(N+1) の事前確率として継承。前ブロックで否定された下位モードの確率は 0% を維持。
- **AT終了時**: モード再抽選。全確率をリセットしデフォルト事前確率を適用。
- **天国例外（実装上）**: 前ブロックで HEAVEN ≥ 50% だったのにAT当選せずCZ失敗 → 「転落」とみなしフルリセット。
- **継承減衰（実装上）**: 誤判定の永続防止のため `inherited = prev * 0.85 + 0.025` の Decay Factor を適用。

# §1.5 解析の前提となるアプリデータ仕様（NotebookLM 解析時の重要前提）

精度向上の主軸として、以下 **2つの観点** で解析を依頼したい。

1. **AT駆け抜け時のモード推定**（後述 §1.5.2）
2. **ゾーン前兆の発生確認による推定精度アップ**（後述 §1.5.1）

両者ともアプリ固有のデータ構造に依存するため、まずその仕様を共有する。

## 1.5.1 「前兆履歴」入力データ仕様（v5.44 で刷新）

ユーザーは各ゾーン (50 / 100 / 150 / 200 / 250 / 300 / 400 / 500 / **即前兆**) で観測した前兆挙動を、以下の **4タイプ + デフォルト未入力** から選択して記録する。

| 内部値 | 表示 | 意味 |
|---|---|---|
| `""` (デフォルト) | **ー** | **ユーザーが当該ゾーンに手を付けていない / 無視した**（= 観測なし） |
| `"前兆"`     | 前兆 | **本前兆**: 規定G数当選を示唆する前兆演出があった |
| `"東京上空"` | 東京上空 | 当該ゾーンで「東京上空」ステージに滞在した（= 前兆ステージ非移行のまま規定Gを通過） |
| `"前兆なし"` | 前兆なし | 前兆ステージに上がらなかった（= 通常ステージのまま規定Gを通過） |
| `"不明"`     | 不明 | **ユーザーが当該ゾーンをウォッチしたが判別不能だった**（= 観測あり・判定失敗） |

### 「ー」と「不明」の意味的区別（最重要）

| 区分 | 観測 | 判定 | 推定への影響 |
|---|---|---|---|
| ー（デフォルト） | **なし** | -    | 補正なし（ベイズ更新スキップ） |
| 不明             | あり     | 失敗 | **何らかの弱い尤度を当てるべき**（現状未定義） |

→ NotebookLM に「不明」タイプの 6モード尤度を提示してほしい（§2.1 各ゾーンの「不明」行を埋めてほしい）。

### ゾーン「即前兆」の特殊性

- AT終了直後 / 通常ブロック開始直後の前兆判定区間（規定G数 50,100… とは別カテゴリ）
- 選択肢は実質「東京上空 / 不明 / ー」のみ（「前兆 / 前兆なし」は意味を持たない）
- → NotebookLM に「即前兆＝○○モード示唆」となる尤度を提示してほしい（§2.1 即前兆ゾーン表を埋めてほしい）

## 1.5.2 「AT駆け抜け」の定義（モード推定への活用余地）

本アプリの AT データ構造において、**1つのATで生成された「SET行」が 1行のみ** で終端する状態を「**AT駆け抜け**」と定義する。

### 構造的定義

- AT中の SET行は「対決 → バイツ獲得」の1サイクルを1行で記録する。
- 対決に勝利した場合、通常はバイツ獲得フェーズに入り、結果として `rowType: "bites"` 行や後続SETが追加される。
- **対決に勝利したがバイツを獲得できなかった場合**、その対決は SET行に記録されるだけで後続の追加行が生成されない。
- 結果として `atEntries[X].rows` が `[{ rowType: "set", ... }]` の 1行のみで終わる状態 = **駆け抜け**。

### 駆け抜けが示唆しうること（解析仮説）

| 仮説 | 推定への活用 |
|---|---|
| バイツ獲得抽選の優遇度がモード/設定に依存 | 駆け抜け発生率からモード/設定をベイズ推定 |
| 駆け抜けはモード再抽選の偏りを誘発する | AT後の `getPrior` を「駆け抜け時はリセットせず○○モード上方修正」に変更 |
| 駆け抜けは弱い当選契機（ゲーム数当選など）と相関 | 当該ブロックの遡及推定（A/B寄りに補正など） |
| 駆け抜け連続は特殊フラグの示唆 | 連続検出時に隠れ状態を仮定する余地 |

### NotebookLM に依頼したい解析

1. **駆け抜けATの発生率が モード / 設定 ごとにどう異なるか**（数値の提示）
2. **駆け抜けAT後のモード再抽選分布に偏りがあるか**（あるなら §3.2 `getPrior` の AT当選後リセット処理を分岐させたい）
3. **駆け抜けATが当該ブロックのモードに対して示唆する内容**（あるなら §1.5.2 を「事後補正ロジック」として実装したい）
4. **駆け抜けが連続した場合の特殊挙動の有無**

→ 数値が出たら、以下のような形式で返却してほしい:

```
§1.5.2 駆け抜け事後補正:
  当該ブロック: A *= 1.3 / B *= 1.2 / C *= 1.0 / CH *= 0.8 / PRE *= 0.6 / HEAVEN *= 0.5
  次ブロック priors override: A=0.50 / B=0.25 / C=0.10 / CH=0.10 / PRE=0.04 / HEAVEN=0.01
```

# §3 推定エンジンのアルゴリズム

`src/lib/engine/modeEstimation.ts` の処理順。

## 3.1 メインループ — `estimateAllModes(blocks)`

各ブロック i について順番に以下を実行し、6モードの確率分布 `ModeProbs` を出力。

```
for i in 0..blocks.length:
  probs = getPrior(i, blocks, results)            // 1. 事前確率
  probs = applyCeilingElimination(probs, jisshuG) // 2. 天井Hard否定
  probs = applyZenchoLikelihood(probs, zencho)    // 3. 前兆履歴ベイズ更新
  probs = applyInvitationConstraints(probs, ...)  // 4. 招待状Hard制約
  probs = applyEyecatch(probs, eyecatch)          // 5. アイキャッチ補正
  probs = normalize(probs)                         // 6. 正規化
  results.push(probs)
```

## 3.2 事前確率 — `getPrior`

```
if i == 0:
  return DEFAULT_PRIORS   // { A:0.40, B:0.20, C:0.10, CH:0.15, PRE:0.10, HEAVEN:0.05 }

if prevBlock.atWin:
  return DEFAULT_PRIORS   // AT当選 → リセット

if prevProbs.HEAVEN >= 0.50:
  return DEFAULT_PRIORS   // 天国例外（実装上の保険）

// 不転落継承 + Decay
for each mode:
  inherited[mode] = prevProbs[mode] * 0.85 + 0.025

// CZ失敗エンドカード → 次ブロック下限適用
if prevBlock.endingSuggestion startsWith "[cz失敗]":
  for mode in CZ_ENDCARD_TABLE[character]:
    inherited[mode] = 0

return normalize(inherited)
```

## 3.3 天井Hard否定 — `applyCeilingElimination`

`jisshuG > MODE_CEILINGS[mode]` なら該当モードを 0 にする。
例: jisshuG=350 なら PRE (天井300) と HEAVEN (天井100) は 0。

## 3.4 前兆履歴ベイズ更新 — `applyZenchoLikelihood`

```
for entry in zencho:                  // entry = "100:東京上空" など
  parse zone & type
  likelihood = ZENCHO_LIKELIHOOD[zone][type]
  if likelihood is null: skip

  // ベイズ更新: posterior ∝ prior × likelihood
  for each mode:
    result[mode] *= likelihood[mode]
  result = normalize(result)
```

## 3.5 招待状Hard制約 — `applyInvitationConstraints`

`eliminateModes` 指定があればそのモードを 0。
`maxRemainingG` 指定があれば `MODE_CEILINGS[mode] > currentG + maxRemainingG` のモードを 0。

## 3.6 アイキャッチ補正 — `applyEyecatch`

- `type: "none"` → スキップ
- `type: "hardConfirm"` → `confirmMode` を 1.0、他を 0 で即返却
- `type: "likelihood"` → ベイズ更新（前兆と同じ乗算+正規化）

## 3.7 正規化 — `normalize`

`total = sum(probs)`、`total <= 0` ならデフォルト事前確率に戻す（救済）。
それ以外は `probs[mode] / total` で確率分布化。
