Claude Code宛: TGRアプリ 抽選格差検証用データ収集レイヤーの追加設計相談
背景
TGRアプリに、運営者(オーナー)が抽選格差の統計的検証を行うための裏側データ収集レイヤーを追加したい。ユーザー側のUI/UXは一切変更せず、既存の入力データから派生する解析用イベントをバックグラウンドで蓄積する設計を検討している。
検証したい仮説
レミニセンス・大食いのリゼ中の押し順ベル契機10.9%抽選(公表値)について、以下を統計的に検証したい:

完全確率10.9%の独立抽選で動いているのか
それとも台・日・セッション・モード等によって**状態差(抽選格差)**が存在し、グロスで10.9%に収束しているだけなのか

検証手法としては、CZ単位の押し順ベル試行数 n と当選数 k のペアを大量に集めて、二項分布の過分散検定、間隔分布の幾何分布適合度検定、可能なら隠れマルコフモデルによる多状態モデルとの比較を行う想定。
設計の制約と方針
1. ユーザーUIは絶対に変更しない

既存の入力フローは完成しており、UX上の崩しは厳禁
新しい入力項目をユーザーに求めることはしない
既存のユーザー入力データから派生・抽出できる情報のみを解析用に蓄積する

2. プラン共通の収集ロジック

無料プランもプロプランも同一の裏側データ収集を行う
プラン別の差はあくまで「ユーザー側に見えるセッション数上限」と「自己集計表示の可否」のみ
解析用データはどちらのプランのユーザーからも見えない

3. 運営者(オーナー)専用の取り出し口

ノリさんだけが集計データにアクセスできる
CSVエクスポート優先(外部解析に投入するため)
Supabase Service Roleキー or 管理者専用ダッシュボードのいずれか

4. 分析結果のフィードバック先

有意な結果が出ればプロプラン限定Discordで発信
これによりプロプランの価値向上にも寄与する設計
ユーザーの個別データを公開することは絶対にしない(統計値のみ)

Claude Codeに確認したい事項(これが今回の核心)
質問1: 現状の記録粒度の確認
最重要: 現在のSupabaseスキーマで、レミニセンスや大食いのリゼ中のゲーム単位の役の成立と当選結果は記録されているか?
具体的には以下の粒度でデータが取れる状態か確認してほしい:

CZ突入1回ごとに、その8G間で

何ゲーム目に
どの役が成立し(押し順ベル / 共通斜めベル / 弱レア役 / 強レア役 / リプレイ / ハズレ)
当選したかしなかったか



もし現状のスキーマがCZ単位の集計値(成功/失敗、合計試行数、最終契機役など)しか持っていないなら、解析に必要な粒度が不足している。その場合、ユーザー入力UIは変えずに、入力イベントをより細かい時系列で記録する方法を検討する必要がある。
質問2: 既存入力フローからの粒度抽出可能性
既存UIでユーザーがCZ中の進行を入力する際、内部的にどこまでのイベントが発生しているか確認してほしい。

ゲームごとのボタン押下イベントは既に発生しているか
それは現状フロントエンドでだけ消費されて捨てられているか、それともバックエンドに送信されているか
既存のテーブルに保存されていないだけで、イベント自体は発生している可能性

もしフロントエンドにはゲーム単位イベントが既に存在するが保存されていないだけなら、既存UI完全維持のまま裏側にイベントロガーを追加すれば済む。これが理想。
質問3: 解析用テーブル設計の提案
仮に既存スキーマに時系列粒度がない前提で、以下のテーブルを別レイヤーとして追加する案を検討してほしい。既存テーブルとは疎結合にし、解析専用として独立させる方針。
提案テーブル
sql-- 解析専用: CZ内の各ゲームイベント
CREATE TABLE analytics_cz_game_events (
  id uuid PRIMARY KEY,
  user_id uuid REFERENCES auth.users(id),
  user_session_id uuid,         -- 既存のユーザーセッション参照
  cz_instance_id uuid,           -- CZ1回を一意に識別
  cz_type text,                  -- 'reminiscence' | 'oogui_rize'
  game_in_cz int,                -- 1〜8
  role text,                     -- 'oshijun_bell' | 'kyoutsu_naname_bell' | 'jaku_rea' | 'kyou_rea' | 'replay' | 'hazure'
  triggered boolean,             -- 当選したか
  is_final_game boolean,         -- リゼでは最終Gで挙動が変わるため必須
  
  -- 状態コンテキスト(あればより強力な解析が可能)
  hall_id_hash text,             -- 店舗ハッシュ(任意、同一店舗判定用)
  machine_id_hash text,          -- 台ハッシュ(任意、同一台判定用)
  is_morning_first boolean,      -- 朝一フラグ
  estimated_setting int,         -- 推測設定(任意)

  pre_cz_invitation text,        -- 直近招待状
  total_g_at_cz_entry int,       -- CZ突入時の総G数(ハマり度合い)
  
  created_at timestamptz DEFAULT now()
);

検討ポイント:

既存の cz_sessions 系テーブルがあるなら、それと重複しないか確認
analytics_* プレフィックスで完全に解析専用と分離するか、既存テーブルを拡張するか
RLS (Row Level Security) で管理者のみREAD可能にする
ユーザー個別の挙動分析を防ぐため、user_id はハッシュ化して保存する設計もあり得る(ただし台×日単位の分析にはハッシュでも問題ない)

質問4: イベント発火点の設計
既存UIの入力ロジックのどこに観測フックを入れるかが論点。以下のどちらが現実的か:
案A: 既存の保存処理に追加トリガー

ユーザーが入力を確定するたびに、既存テーブルへのinsertと並行してanalytics_*テーブルにもinsert
シンプルだが、既存ロジックへの侵入度が高い

案B: 既存テーブルへのDBトリガーで派生

Supabase側で既存テーブルへのinsert/update時にanalytics_*テーブルへ自動派生
アプリコードへの侵入なし、ただし派生に必要な情報が既存テーブルに揃っている必要あり

案C: 専用のロガー関数を一箇所追加

アプリ内にlogAnalyticsEvent()関数を作り、既存の入力確定箇所から呼び出すだけ
影響範囲が明確、既存ロジックを壊さない

僕の推奨は案C。既存テーブルに必要情報が揃っていなくても柔軟に拡張できる。
質問5: 管理者向けデータ取得の実装
ノリさんが解析用データを取り出すための仕組み。以下のいずれか:
案1: 管理者専用ダッシュボード(/admin/analytics)

Supabase認証で is_admin フラグ持つユーザーのみアクセス可
CSVダウンロードボタン、簡易な集計表示
実装コストはやや高い

案2: Supabase Service Roleキー直接利用

ノリさんがSQL Editor or psqlから直接クエリ
実装コストゼロ、ただし運用は技術前提

案3: 定期バッチでGoogle Sheets / Discord連携

日次/週次で集計をGoogle Sheetsに書き出し
ノリさんはSheetsで確認、必要時にCSV化

ノリさんはGoogle Sheets熟練者なので、案3 + 案2の併用が現実解な気がしている。Claude Codeから別案あれば歓迎。
質問6: プライバシーポリシーの追記
裏で解析用データを収集する以上、利用規約・プライバシーポリシーに明記が必要。以下の追記を想定:

収集データ: ユーザーの実戦記録(役の成立、CZ結果等)から派生する統計用データ
収集目的: 機種抽選挙動の統計的検証、サービス改善
個人特定情報は含まない、個別データを運営者が閲覧することはない(統計集計のみ)
オプトアウト: 設定画面から拒否可能(実装する場合)

既存LP/利用規約への組み込み方も合わせて提案してほしい。
まず確認してほしいこと(優先順位)

質問1: 現状のスキーマでゲーム単位の粒度が取れているか確認 ← 最優先
質問2: 既存入力フローのフロントエンドイベント発生粒度の確認
質問3: 解析テーブル設計案の妥当性検討
質問4: イベント発火点の設計方針確定

質問1と2の確認結果次第で、必要な実装範囲が大きく変わる(既にイベントが発生していれば追加実装は最小、なければスキーマと記録ロジックの両方が必要)。
